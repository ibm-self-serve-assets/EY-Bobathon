1 Current Application Architecture

The application is a single-project ASP.NET Web API hosted on IIS via System.Web, targeting .NET Framework 4.8. It uses no DI container — all services are instantiated directly via new inside controllers and other services.

4 Tax Calculation Flow

Performed by TaxCalculationService.Calculate(TaxFiling), exposed via GET api/taxfilings/{id}/calculation.

TaxFiling
    │
    ▼
TaxableIncome = AnnualIncome − DeductionAmount  (min 0)
    │
    ▼
EstimatedTax = CalculateEstimatedTax(TaxableIncome, FilingType)
    │  Individual:  ≤ 100,000 → 10%
    │              ≤ 500,000 → 20%
    │              > 500,000 → 30%
    │  Partnership: flat 24%
    │  Corporate (default): flat 21%
    │
    ▼
EffectiveTaxRate = EstimatedTax / AnnualIncome  (0 if income = 0)
    │
    ▼
RiskBand = CalculateRiskBand(filing, taxableIncome, effectiveRate)
    │  DeductionAmount > AnnualIncome × 30% → "High"
    │  TaxableIncome > 500,000 OR EffectiveRate < 10% → "Medium"
    │  Otherwise → "Low"
    │
    ▼
ReviewRecommendation
    │  "High"   → "Enhanced review required"
    │  "Medium" → "Standard review required"
    │  "Low"    → "Auto-approval candidate"
    │
    ▼
TaxCalculationResult (DTO returned to caller)
Important: The calculation is stateless and on-demand — it does not persist the computed RiskBand back to the filing. The RiskBand on TaxFiling is a separate field set at creation time.
Performance note: DashboardService.GetSummary() calls Calculate() for every filing to sum TotalEstimatedTax. This is an N-call loop per dashboard request — acceptable now, a bottleneck once a real database is introduced.

5 Validation Logic

Validation is implemented in TaxValidationService and called from TaxFilingService.Validate().

Filing Validation Rules

RuleError Message
filing == null"Tax filing payload is required."
TaxPayerName is null/empty"Taxpayer name is required."
TaxYear is null/empty"Tax year is required."
AnnualIncome < 0"Annual income cannot be negative."
DeductionAmount < 0"Deduction amount cannot be negative."
DeductionAmount > AnnualIncome"Deduction amount cannot exceed annual income."
FilingType is null/empty"Filing type is required."

Review Status Validation

Accepts only: "Draft", "Under Review", "Approved", "Rejected".

Validation Gaps — Missing Rules

  • No validation that TaxPayerId references an existing TaxPayer.
  • No validation of TaxYear format (e.g. 4-digit year only).
  • No validation of FilingType against the allowed value set.
  • TaxPayersController.CreateTaxPayer performs no validation beyond a null check.
  • No duplicate detection — the same taxpayer + tax year combination can be filed multiple times.

8 Migration Risks and Blockers

🔴 Blockers

#RiskDetail
B1 System.Web dependency The entire hosting model (System.Web.HttpApplication, IIS/System.Web pipeline) is incompatible with .NET 8. All controllers and hosting infrastructure must be rewritten from scratch.
B2 Static mutable shared state InMemoryTaxRepository uses static readonly List<T> fields. Must be refactored into a Singleton-registered instance class — left as-is it is untestable and not thread-safe.
B3 No DI container All service dependencies are newed up manually, making unit testing impractical. Introducing DI is a prerequisite for clean .NET 8 code.

🟡 Risks

#RiskDetail
R1Newtonsoft.Json vs System.Text.Json.NET 8 defaults to System.Text.Json. Property naming conventions, null handling, and circular reference behaviour differ. Decision required before migration.
R2Synchronous I/OAll service methods are synchronous. Minimal API strongly favours async Task<IResult>. Mixing sync/async can cause thread-pool starvation under load.
R3N-call calculation in DashboardDashboardService.GetSummary() calls Calculate() for every filing. Acceptable with in-memory data; becomes a bottleneck with a real database.
R4No authentication / authorisationNo auth middleware exists anywhere. RequireAuthorization() is easy to add in .NET 8 but policy design must be defined first.
R5Denormalised TaxPayerNameTaxFiling.TaxPayerName duplicates TaxPayer.TaxPayerName. If a taxpayer is renamed, filings become stale. No FK enforcement exists.
R6String-typed enumsReviewStatus, FilingType, RiskBand, EventType are all free-text strings with no compile-time safety. Changing to enum is a breaking API change.
R7Thread safety of static listsList<T> is not thread-safe. Concurrent POST requests can corrupt in-memory lists. A Singleton repository needs ConcurrentBag or locks.
R8No error handling middlewareUnhandled exceptions return raw 500 responses with stack traces in debug mode. app.UseExceptionHandler() or a Results.Problem() wrapper must be added.

9 Recommended Migration Sequence

Minimises risk by migrating infrastructure first, then business logic, then endpoints.

1
Project Bootstrap
  1. Create a new SDK-style .csproj targeting net8.0
  2. Create Program.cs with WebApplication.CreateBuilder()
  3. Replace Web.config with appsettings.json
  4. Add NuGet packages: Microsoft.AspNetCore.OpenApi, optionally Swashbuckle.AspNetCore
2
Models and DTOs — Reuse
  1. Copy all files from Models/ and DTOs/ with namespace updated to the new project
  2. No logic changes required in either layer
3
Service Layer Refactor — Introduce DI
  1. Convert InMemoryTaxRepository from static to an instance class; register as Singleton
  2. Inject InMemoryTaxRepository into all services via constructor injection
  3. Inject TaxValidationService and AuditService into the services that depend on them
  4. Register all services in Program.cs via builder.Services.AddSingleton / AddScoped
  5. TaxCalculationService and TaxValidationService require no changes beyond namespace
4
Minimal API Endpoints
  1. Replace each controller action with an app.Map*() call — one endpoint extension file per domain area
  2. Replace IHttpActionResult returns with Results.Ok(), Results.NotFound(), Results.BadRequest()
  3. Remove all [FromBody] attributes — implicit in Minimal API
5
Serialisation and Error Handling
  1. Decide: keep System.Text.Json (recommended) or add AddNewtonsoftJson()
  2. Add global exception handling via app.UseExceptionHandler() or Results.Problem()
6
Validation and API Hardening
  1. Add FluentValidation or DataAnnotations for request validation in Minimal API
  2. Consider strongly-typed enums or constants for ReviewStatus, FilingType, RiskBand
7
OpenAPI / Swagger
  1. Add AddOpenApi() and configure Swagger UI for endpoint documentation
  2. Add .WithName() and .WithSummary() to each endpoint

11 Assumptions and Missing Information

#ItemDetail
A1No unit testsNo test project was found in the repository. There is no automated safety net for the migration. Characterisation tests are strongly recommended before migrating.
A2No authenticationNo auth layer exists anywhere. Analysis assumes auth will be added as a separate concern post-migration.
A3In-memory data is intentionalThe static repository appears to be a deliberate demo/workshop choice. A production migration would replace this with EF Core or another real data layer.
A4TaxPayerId is not enforcedFilings reference a TaxPayerId but no validation confirms the taxpayer exists. This data integrity gap exists in the legacy code and would carry over as-is.
A5FilingType is not validated against allowed valuesThe calculation engine uses "Individual" and "Partnership", and falls back to a corporate rate for all other values — silently.
A6No CORS, no HTTPS configurationWeb.config has no HTTPS or CORS settings. Both must be explicitly configured in .NET 8.
A7No loggingNo ILogger or logging framework is used anywhere. Built-in structured logging via ILogger<T> should be injected into services during migration.
A8Thread safety not testedThe static List<T> is not thread-safe. Not verified as a bug in this demo context, but a real risk in any concurrent production scenario.
